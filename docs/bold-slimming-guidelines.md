# Bold Slimming Guidelines（內容粗體瘦身規範）

目標：讓粗體更「有理由」——讀者一眼就知道粗體是在標示**清單項目名稱/關鍵詞**，而不是到處畫重點。

## 適用範圍（SSOT）

- `docs/src/content/attractions.*.md`
- `docs/src/content/stays.*.md`
- `docs/src/content/transport.*.md`
- `docs/src/content/extensions.*.md`
- `docs/src/content/itinerary.md`
- `docs/src/content/misc.md`
- `docs/src/content/README.md`

不處理：

- `docs/src/content/*.legacy.md`

## 核心規則（唯一保留粗體的理由）

只保留「清單領頭（list lead）」的粗體，用來標出**項目名稱 / 主要關鍵詞**。

```md
- **A Vida Portuguesa**：...
- **[Cutipol](...)**：...
- **罐頭＝儀式感（送禮向）**：...
```

判斷法（簡單版）：

- **保留**：如果不先看到「名字」，讀者會不知道這一條在講什麼。
- **移除**：如果拿掉粗體後，整句仍然自然順讀（只是少了螢光筆），那就不該用粗體。

## 一律移除/降級的粗體類型

### 1) 段落內純強調粗體 → 改一般文字

```md
Before:
... **它相對平坦**，走起來很輕鬆。
... **各買 1 顆趁熱吃**，外酥內軟。

After:
... 它相對平坦，走起來很輕鬆。
... 各買 1 顆趁熱吃，外酥內軟。
```

### 2) 用粗體包住術語/外語/指令 → 改成反引號

原則：

- **術語/外語/短指令**：用反引號（`...`）表示「這是一個詞/指令」。
- **一般形容/強調**：不要用粗體。用內容把畫面與理由講清楚（先介紹→再推薦），讀者自然會記住重點。

```md
Before:
咖啡：點 **bica** / **galão** / **com espuma**
交通：在機器上選 **bilhete**

After:
咖啡：點 `bica` / `galão` / `com espuma`
交通：在機器上選 `bilhete`
```

### 3) 標題行裡的粗體 → 一律移除

標題本身已經是視覺重點，不要再疊粗體。

```md
Before:
## **Coffee 101**

After:
## Coffee 101
```

## 建議寫法（讓重點自然，不靠粗體）

- **把「很推薦」寫成「為什麼值得」**：味道、場景、時間帶、人潮、定位、氛圍、你會記住的那個瞬間。
- **用結構替代粗體**：前半句先交代情境/迷人點，後半句再給建議（或推薦品項）。
- **補充資訊用括號**：把「備註/限制/以現場為準」放括號，不要用粗體硬畫重點。

## 快速自檢清單（改完每篇必做）

- **list lead**：粗體只出現在清單項目領頭（`- **...**：` 或 `- **[...](...)**：`）。
- **段落**：段落內不使用粗體做強調（目標是 0）。
- **術語**：術語/外語/短指令改成反引號（`...`），且一致。
- **標題**：任何 `##/###/####` 標題行不含 `**`。

## UI / Parser 敏感寫法（避免 silent wrong / missing）

這份 repo 的 UI 端不會用完整 Markdown 引擎渲染內容，而是用較保守的解析器（例如 `FormattedText`、`RichContent`）。有些寫法在一般 Markdown 看起來沒問題，但 UI 可能解析不到或渲染怪異。

### 1) 行內 `**` 必須成對

```md
Do:
… 這裡可以點 **Bica**。

Don’t:
… 這裡可以點 **Bica。
```

### 2) `{{gallery:...}}` token 不要被拆壞

- **Do**：把 `{{gallery:Title|url1|url2}}` 完整放在同一行，不要換行、不要在 token 內插入其他 Markdown。
- **Don’t**：把 token 拆行，或在 token 旁邊硬塞粗體/反引號，導致圖片區塊消失。

### 3) `@card:` 必須是「單獨一行的摘要」

某些 UI 會把段落切成 `###` 小節後做 carousel 卡片摘要；摘要行格式是：

```md
Do:
@card: 這是一句摘要

Don’t:
@card:
```

補充規範（讓它像摘要，而不是第二段內文）：

- **放哪裡**：固定放在 `### ...` 的下一行；下一行才是 `{{gallery:...}}` 或正文。
- **寫什麼**：一句話回答「這個小節給你什麼回報？」（視角、體驗、看點、風格、風險或定位）。盡量寫「結果」而不是「方法」，避免攻略口吻（例如一直在講怎麼省力/怎麼避隊）。
- **建議長度**：50–90 個中文字左右。太短像標籤、資訊不足；太長會變成第二段正文，卡片上反而難掃讀。
- **格式禁忌**：
  - 不要用粗體當重點（`@card:` 本來就是 UI 重點）。
  - 反引號只包詞（例如 `15E`、`bica`），不要整句都包起來。
  - 不要在 `@card:` 行內塞 `{{gallery:...}}`，也不要把 gallery token 拆行。

### 4) 反引號只包「詞」，不要包整句

```md
Do:
點 `bica` / `galão`。

Don’t:
`點 bica / galão 就好，然後…`
```
